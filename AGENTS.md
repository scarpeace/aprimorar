# AGENTS.md

Operational guide for coding agents working in the Aprimorar repository.

## Project Snapshot
- Monorepo with a React 19 + TypeScript + Vite SPA in `client/`
- Spring Boot 3.5 + Java 21 API in `server/api-aprimorar/`
- PostgreSQL in development, with Flyway migrations under `server/api-aprimorar/src/main/resources/db/migration/`
- User-facing text, validation messages, and API error messages must be written in Portuguese
- Frontend API hooks, types, and schemas are generated by Kubb into `client/src/kubb/`

## Repository Layout
- `client/src/features/` contains route pages and feature-specific UI
- `client/src/components/ui/` contains shared UI primitives
- `client/src/lib/` contains utilities, shared helpers, and client-side infrastructure
- `server/api-aprimorar/src/main/java/com/aprimorar/api/domain/` contains backend business modules
- `server/api-aprimorar/src/test/java/` contains JUnit and Mockito tests
- `client/AGENTS.md` and `server/api-aprimorar/AGENTS.md` add more specific rules for each app

## Agent Workflow
- Keep diffs scoped to the task; avoid opportunistic refactors
- Read the closest `AGENTS.md` before editing files inside nested app folders
- Preserve the formatting style already used in touched files; this repo is not fully normalized
- Never hand-edit generated files under `client/src/kubb/`; regenerate them instead
- For backend persistence changes, update migration, entity, DTOs, mapper, service, tests, and frontend consumers together
- Do not rename API fields, enum values, or endpoints unless backend and frontend are updated in the same change
- Prefer fixing root causes over adding casts or one-off workarounds that fight generated contracts

## Build And Run Commands

### Frontend (`client/`)
```bash
npm install
npm run dev
npm run lint
npm run build
npm run preview
npm run sync
```

- `npm run build` runs `tsc -b && vite build`
- `npm run sync` regenerates Kubb artifacts from the backend OpenAPI document
- There is currently no dedicated frontend unit test script in `client/package.json`

### Backend (`server/api-aprimorar/`)
```bash
docker compose up -d db
docker compose down
./mvnw spring-boot:run
./mvnw test
./mvnw verify
./mvnw package
./mvnw -Pgenerate-openapi generate-resources
```

- Spring Boot Docker Compose support may wire local services when the API starts
- The OpenAPI generation profile writes `openapi.json` under `target/generated/openapi/`

### Frontend verification

## Change Verification
- Frontend-only change: run `npm run lint` and `npm run build` in `client/`
- Backend-only change: run a targeted Maven test first, then `./mvnw test` when practical
- Full-stack change: run backend tests, then frontend build; regenerate Kubb if the API contract changed
- If backend request/response models used by the SPA change, run the API and then `npm run sync` in `client/`
- If generated Kubb schemas fail at runtime, inspect the backend DTO/OpenAPI contract before weakening client validation

## Frontend Guidance

### Imports and modules
- Import order: React/core, third-party, `@/` aliases, then relative imports
- Use `import type` for type-only imports
- Prefer `@/` aliases for cross-feature imports; keep relative imports within the same feature when convenient
- Use named exports for components, hooks, and helpers; avoid `export default`

### Components and files
- Route pages use `*Page.tsx` naming
- Feature-specific UI lives under `src/features/<feature>/components/`
- Shared reusable UI belongs in `src/components/ui/`
- Keep route components thin; move reusable logic into hooks, helpers, or subcomponents
- Prefer early returns for loading, empty, and error states instead of deep nesting

### Types and validation
- Use Zod schemas and inferred types at API/form boundaries
- Treat generated Kubb types and schemas as the source of truth for API contracts
- Prefer explicit prop interfaces or types near the top of the file
- React Hook Form should use `zodResolver` and usually `mode: "onBlur"`
- Avoid `as unknown as ...` unless there is no safer option and the mismatch is fully understood

### State and data fetching
- Use TanStack Query for server state
- Reuse stable query keys; avoid ad hoc string keys
- Use `enabled` to guard queries that depend on route params or other async inputs
- Use placeholder data or `keepPreviousData` for paginated views when it matches existing patterns
- Invalidate related queries after successful mutations
- Reset local pagination/search state in event handlers when possible instead of effect-based setState cascades

### Formatting and naming
- Follow the style already present in each file; some files use semicolons and some do not
- Do not mass-reformat unrelated lines just to normalize style
- Components, pages, DTO-like types, and enums use PascalCase
- Hooks use `useXxx`; functions and variables use camelCase
- CSS module files use PascalCase when that pattern already exists nearby

### Error handling
- Prefer user-friendly Portuguese messages in UI and validation copy
- Surface API failures through dedicated UI states instead of silent `console` handling
- Use shared helpers such as `getFriendlyErrorMessage` when the file already follows that pattern
- Validate external data with Zod when it crosses the network boundary

## Backend Guidance

### Architecture and layering
- Controllers should stay focused on HTTP concerns only
- Services own business rules and transaction boundaries
- Repositories handle persistence and query definitions
- Mappers convert DTOs to and from entities; do not leak entities into controllers
- Domain packages commonly include `dto/`, `repository/`, `web/`, and `exception/`

### Java conventions
- Prefer constructor injection
- DTOs are typically Java records with Bean Validation annotations
- Use `@Transactional(readOnly = true)` for queries and `@Transactional` for commands
- Use UUID identifiers and shared timestamp fields from `BaseEntity`
- Prefer helper methods such as `findXOrThrow` and `resolveXOrThrow`

### Contracts and dates
- API DTO field types must stay consistent with generated frontend schemas
- If the generated client rejects backend payloads, fix the contract or mapper rather than weakening generated parsing by hand
- Be explicit about timezone strategy at API boundaries; if request/response models differ from persistence types, convert in mappers

### Naming and validation
- Classes use PascalCase; methods and fields use camelCase
- Exception classes should be domain-specific and end with `Exception`
- Validation and business-rule messages must be Portuguese
- Repository query methods should stay explicit and intention-revealing

### Error handling and API behavior
- Centralize HTTP error shaping in `GlobalExceptionHandler`
- Prefer `ProblemDetail` responses with stable custom `code` properties
- Use status codes consistently: `200` reads/updates, `201` creates, `204` deletes/archive actions, `400` invalid input, `404` missing resources, `409` conflicts
- Throw domain exceptions instead of returning null or swallowing invalid states

### Testing
- Tests use JUnit 5, AssertJ, and Mockito
- Organize service tests with `@Nested` classes when it improves readability
- Use deterministic fixtures with fixed UUIDs and timestamps
- Prefer targeted Maven runs while iterating; expand to broader verification before finishing

## Generated Code And Contracts
- `client/src/kubb/` is generated; regenerate instead of editing hooks, schemas, or types manually
- If backend DTOs, enums, or endpoint signatures change, regenerate OpenAPI and Kubb in the same task
- When type mismatches appear between generated schemas and handwritten code, prefer adapting handwritten abstractions to the generated contract
