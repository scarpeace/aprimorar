# AGENTS.md

Operational guide for agentic coding agents working in this repository.

## Scope
- This file applies to the whole workspace at `/home/scarpellini/Documents/Projetos/aprimorar`.
- There are more specific guides in `client/AGENTS.md` and `server/api-aprimorar/AGENTS.md`.
- When working inside `client/`, follow the root guide plus `client/AGENTS.md`.
- When working inside `server/api-aprimorar/`, follow the root guide plus `server/api-aprimorar/AGENTS.md`.
- Prefer minimal, targeted changes. Do not reformat unrelated files.

## Repository Layout
- `client/`: React 19 + TypeScript + Vite SPA.
- `server/api-aprimorar/`: Spring Boot 3.5 + Java 21 API.
- Root `package.json`: developer convenience scripts for starting backend and frontend together.
- Frontend API hooks and schemas are generated by Kubb from the backend OpenAPI document.

## Working Norms
- Inspect existing code before editing; match local patterns instead of inventing new ones.
- Keep user-facing strings and backend validation/business messages in Portuguese.
- Do not manually edit generated frontend files under `client/src/kubb/`.
- If backend request/response contracts change, regenerate OpenAPI and then regenerate the Kubb client.
- Prefer focused verification for the area you changed, then broaden if the change crosses frontend/backend boundaries.

## Root Commands
Run from the repository root unless noted otherwise.

- Install root helper deps: `npm install`
- Start backend only: `npm run start:backend`
- Start frontend only after backend docs are up: `npm run start:frontend`
- Start full stack in parallel: `npm run dev`
- Start frontend only without backend wait logic: `npm run dev:frontend`

## Frontend Commands
Run from `client/`.

- Install deps: `npm install`
- Dev server: `npm run dev`
- Build and type-check: `npm run build`
- Lint: `npm run lint`
- Preview production build: `npm run preview`
- Regenerate API client from OpenAPI: `npm run sync`

## Frontend Test Commands
- There is currently no dedicated frontend test runner configured in `client/package.json`, so there is no supported single-test command.
- For frontend work, the minimum expected verification is usually `npm run lint` and `npm run build`.

## Backend Commands
Run from `server/api-aprimorar/`.

- Start local database: `docker compose up -d db`
- Stop local database: `docker compose down`
- Run the API locally: `./mvnw spring-boot:run`
- Run all tests: `./mvnw test`
- Build package: `./mvnw package`
- Full verification with JaCoCo: `./mvnw verify`
- Generate OpenAPI document: `./mvnw -Pgenerate-openapi generate-resources`

## Running a Single Backend Test
Use focused Maven runs while iterating.

- Single test class: `./mvnw -Dtest=ParentServiceTest test`
- Single test method: `./mvnw -Dtest=ParentServiceTest#shouldReturnPagedParents test`
- Pattern match: `./mvnw -Dtest='*ServiceTest' test`

## Full-Stack Contract Workflow
- Start the database if needed: `docker compose up -d db` from `server/api-aprimorar/`.
- Start the backend and ensure `http://localhost:8080/v3/api-docs` is available.
- Regenerate backend OpenAPI with `./mvnw -Pgenerate-openapi generate-resources`, then regenerate frontend hooks/types with `npm run sync` from `client/`.
- After regeneration, verify the frontend still passes `npm run build`.

## Formatting and Style Baseline
- Follow the formatting already present in the touched file.
- Frontend code mostly uses double quotes and semicolons.
- Backend Java uses 4-space indentation and conventional Spring formatting.
- Prefer ASCII unless the file already uses another character set; Portuguese content is expected where user-visible.

## Import Conventions
- Keep imports grouped in a stable order.
- Frontend order: React/core, third-party packages, `@/` aliases, then relative imports.
- Use `import type` for type-only frontend imports.
- Backend imports should remain organized by the formatter/style already used in each file.

## Naming Conventions
- Frontend components, pages, records, and exported types use `PascalCase`.
- Frontend hooks use `useXxx`.
- Frontend variables, functions, props, and object fields use `camelCase`.
- Backend classes use `PascalCase`.
- Backend methods and fields use `camelCase`.
- Prefer descriptive names over abbreviations unless an abbreviation is already standard in the surrounding code.
- Route page files in the SPA commonly use `*Page.tsx`, and backend DTOs commonly use names like `StudentCreateDTO`, `StudentUpdateDTO`, and `StudentResponseDTO`.

## Frontend Architecture
- Keep feature-specific code under `client/src/features/`.
- Keep shared UI primitives under `client/src/components/ui/`.
- Keep shared utilities and infrastructure under `client/src/lib/`.
- Parent pages should usually own search, pagination, filters, and query orchestration.

## Frontend Types and Validation
- TypeScript is in `strict` mode; do not bypass types casually.
- Prefer explicit prop types near the top of the file.
- Use Zod schemas and inferred types at API/form boundaries.
- Prefer `z.input<typeof schema>` for form inputs when transforms or coercions are involved.
- React Hook Form is commonly configured with `zodResolver(...)` and `mode: "onBlur"`.
- Treat generated Kubb hooks/types/schemas as the source of truth for backend contracts.
- Avoid unsafe type assertions unless the mismatch is understood and unavoidable.

## Frontend State and Data Fetching
- Use TanStack Query for server state.
- Reuse existing generated query keys and hooks instead of inventing ad hoc fetch layers.
- Use `enabled` for queries that depend on optional route params or other conditional inputs.
- Prefer early returns for loading, missing-data, and error states.

## Frontend Error Handling
- Surface API failures in the UI; do not leave failures as `console.log` only.
- Prefer shared helpers such as `getFriendlyErrorMessage` when they fit the pattern already used nearby.
- Keep user-facing error messages in Portuguese.
- If generated schemas reject backend data, investigate the backend contract before weakening frontend validation.

## Backend Architecture
- Domain packages live under `com.aprimorar.api.domain.<entity>`.
- Common backend package patterns include `dto`, `repository`, `exception`, and controller/service/mapper classes in the domain package.
- Controllers own HTTP concerns only.
- Services own business rules and transaction boundaries.
- Mappers translate between entities and DTOs.
- Keep JPA entities from leaking directly to API responses.

## Backend Spring and JPA Conventions
- Prefer constructor injection.
- DTOs are commonly Java records with Bean Validation annotations.
- Use `@Transactional(readOnly = true)` for reads and `@Transactional` for writes.
- Keep helper methods such as `findXOrThrow` private inside services when they are local details.
- Be intentional about fetch behavior and entity relationships.

## Backend HTTP and Error Handling
- Keep request parsing and validation annotations in controllers; keep business validation in services.
- Use stable status codes and avoid casual endpoint or field renames.
- Throw domain-specific exceptions instead of returning `null` or generic failures.
- Centralize error response shaping in `GlobalExceptionHandler`.
- Keep validation and business-rule messages actionable and in Portuguese.
- Prefer maintaining stable error codes and response shapes for frontend consumers.

## Backend Testing Guidance
- Tests use JUnit 5, AssertJ, and Mockito.
- Prefer deterministic fixtures with fixed UUIDs and timestamps.
- Use `@Nested` test classes when they make scenarios clearer.
- If DTOs, mappers, or service rules change, update the relevant tests in the same task.

## Change-Specific Expectations
- Backend persistent field change: update entity, DTOs, mapper, service logic, tests, and Flyway migration if required.
- Backend API contract change: update controller/docs, DTOs, tests, OpenAPI generation, and frontend Kubb sync.
- Frontend work touching generated contracts: do not patch generated code by hand unless the user explicitly wants a temporary workaround.
- Cross-stack changes should usually be verified with backend tests plus frontend lint/build.

## Files and Areas to Treat Carefully
- Do not manually edit `client/src/kubb/`.
- Be cautious around backend exception handling and DTO contracts because the SPA depends on them.
- Keep generated or tool-managed files limited to the commands that own them.
