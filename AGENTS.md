# AGENTS.md

Operational guide for coding agents working in the Aprimorar repository.

## Project Snapshot
- React 19 + TypeScript + Vite SPA in `client/`
- Spring Boot 3.5 + Java 21 API in `server/api-aprimorar/`
- PostgreSQL for development; Flyway migrations in `server/api-aprimorar/src/main/resources/db/migration/`
- User-facing copy, validation, and API error messages must be written in Portuguese
- Frontend API types/hooks are generated by Kubb into `client/src/kubb/`

## Repository Layout
- `client/src/features/` contains route-level feature code
- `client/src/components/ui/` contains shared UI primitives
- `client/src/lib/` contains shared utilities and helpers
- `server/api-aprimorar/src/main/java/com/aprimorar/api/domain/` contains backend business modules
- `server/api-aprimorar/src/test/java/` contains JUnit and Mockito tests

## Agent Workflow
- Keep diffs scoped; avoid drive-by refactors
- Check the closest `AGENTS.md` before editing inside `client/` or `server/api-aprimorar/`
- Preserve existing local style in touched files; this repo is not fully format-normalized
- Do not hand-edit generated code in `client/src/kubb/`; regenerate it instead
- For persisted backend changes, update the migration, entity, DTOs, mapper, service, and any frontend consumers together
- Do not rename API fields or endpoints without aligning backend and frontend in the same change

## Build And Run Commands

### Frontend (`client/`)
```bash
npm install
npm run dev
npm run lint
npm run build
npm run preview
npm run sync
```

- `npm run build` runs `tsc -b && vite build`
- `npm run sync` runs Kubb code generation from the backend OpenAPI document
- There is currently no dedicated frontend test script in `client/package.json`

### Backend (`server/api-aprimorar/`)
```bash
docker compose up -d db
docker compose down
./mvnw spring-boot:run
./mvnw test
./mvnw verify
./mvnw package
./mvnw -Pgenerate-openapi generate-resources
```

- Spring Boot Docker Compose support is enabled; starting the API may also wire local services automatically
- The OpenAPI generation profile writes `openapi.json` under `target/generated/openapi/`

### Frontend verification
- No unit test runner is configured right now
- Use `npm run lint` for static checks

## Change Verification
- Frontend-only change: run `npm run lint` and `npm run build` in `client/`
- Backend-only change: run targeted `./mvnw -Dtest=... test`; finish with `./mvnw test` when practical
- Full-stack change: run backend tests, then frontend build; regenerate Kubb artifacts if the API contract changed
- If you change backend request/response models used by the SPA, run the API and then `npm run sync` in `client/`

## Frontend Style Guide

### Imports and modules
- Import order: React/core, third-party packages, `@/` aliases, then relative imports
- Use `import type` for type-only imports
- Prefer `@/` aliases for cross-feature imports and relative paths within the same feature
- Use named exports for components, hooks, and helpers; avoid `export default`

### Components and files
- Route pages use `*Page.tsx` naming
- Feature-specific UI stays under `src/features/<feature>/components/`
- Shared primitives belong in `src/components/ui/`
- Keep page components thin; push reusable logic into hooks/helpers
- Prefer early returns for loading, empty, and error states

### Types and validation
- Use Zod schemas and inferred types for API/form boundaries
- Prefer explicit prop interfaces near the top of the file
- React Hook Form should use `zodResolver` and `mode: "onBlur"`
- Treat generated Kubb schemas/types/hooks as the source of truth for API contracts

### State and data fetching
- Use TanStack Query hooks for server state
- Centralize or reuse stable query keys; avoid ad hoc string keys
- Use `keepPreviousData` or placeholder data for paginated views when it fits existing patterns
- Invalidate relevant queries after successful mutations

### Formatting and naming
- Follow the formatting already used in the file you touch; some files use semicolons and some do not
- Do not mass-reformat unrelated lines just to normalize style
- Components, pages, and types use PascalCase
- Hooks use `useXxx` names
- Variables and functions use camelCase
- CSS modules use PascalCase file names when that pattern already exists

### Frontend error handling
- Prefer user-friendly Portuguese messages
- Surface API failures through shared UI states instead of silent `console` handling
- Validate external data with Zod when it crosses the network boundary

## Backend Style Guide

### Architecture and layering
- Keep controllers focused on HTTP concerns only
- Put business rules and transaction boundaries in services
- Keep repositories for persistence and query definitions
- Use mappers for DTO <-> entity conversion
- Domain modules commonly include `dto/`, `repository/`, `web/`, and `exception/`

### Java conventions
- Prefer constructor injection
- DTOs are typically Java records with Bean Validation annotations
- Use `@Transactional(readOnly = true)` for query methods and `@Transactional` for commands
- Use UUID identifiers and shared base-entity fields from `BaseEntity`
- Prefer helper methods such as `findXOrThrow` and `resolveXOrThrow`

### Naming and validation
- Classes use PascalCase; methods and fields use camelCase
- Exception classes should be domain-specific and end with `Exception`
- Validation and business-rule messages must be Portuguese
- Repository query methods should stay explicit

### Error handling and API behavior
- Centralize HTTP error shaping in `GlobalExceptionHandler`
- Prefer `ProblemDetail` responses for API errors, with stable custom `code` properties
- Use status codes consistently: `200` for reads/updates, `201` for creates, `204` for deletes/archive actions, `400` for invalid input, `404` for missing resources, `409` for conflicts
- Throw domain exceptions instead of returning null or swallowing invalid states

### Testing
- Tests use JUnit 5, AssertJ, and Mockito
- Organize service tests with `@Nested` classes for query vs command behavior when helpful
- Use deterministic fixtures with fixed UUIDs and timestamps
- Prefer focused single-class or single-method Maven runs while iterating

## Generated Code And Contracts
- `client/src/kubb/` is generated by Kubb from the backend OpenAPI spec
- Regenerate instead of editing generated hooks, schemas, or types manually
- If backend DTOs or endpoints change, verify the generated client still compiles after `npm run sync`
