<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → `/v1/auth/login` | untrusted credential input enters authentication flow |
| browser cookie/session → `/v1/**` | authenticated state crosses from client-held cookie into protected API |
| frontend dev origin → backend API | cross-origin credentialed requests depend on correct CORS and CSRF handling |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | S | `POST /v1/auth/login` | mitigate | Use Spring Security authentication with BCrypt hash comparison and username-or-email lookup through `InternalUserDetailsService`; reject inactive users. |
| T-01-02 | T | session-backed write endpoints | mitigate | Keep CSRF enabled with `CookieCsrfTokenRepository.withHttpOnlyFalse()` so cookie-authenticated writes require CSRF header validation. |
| T-01-03 | I | protected `/v1/**` controllers | mitigate | Require authentication for all `/v1/**` endpoints except explicit auth/docs paths so anonymous users cannot read business data. |
| T-01-04 | E | CORS/session configuration | mitigate | Restrict allowed origin to `http://localhost:5173`, enable credentials deliberately, and return JSON unauthorized responses through centralized handlers. |
</threat_model>
