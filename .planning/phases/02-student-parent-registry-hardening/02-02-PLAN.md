<interfaces>
Current student API shape:

```java
public record StudentRequestDTO(..., UUID parentId) {}
public record StudentResponseDTO(..., UUID parentId, Instant archivedAt, Instant createdAt, Instant updatedAt) {}
```

Keep the write contract simple for the existing flow, but make read responses easier for the SPA to render directly:

```java
public record StudentRequestDTO(..., UUID parentId) {}
public record StudentResponsibleSummaryDTO(UUID responsibleId, String name, String contact, String cpf) {}
public record StudentResponseDTO(..., UUID parentId, StudentResponsibleSummaryDTO responsible, ...) {}
```

Keep the term `Responsável` in user-visible messages per D-10, preserve the single registration flow per D-01, and do not add many-to-many, primary, or financial semantics per D-05.
</interfaces>
</context>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → student/responsável write endpoints | untrusted responsible IDs enter the API |
| parent archive/delete commands → linked students | destructive lifecycle actions can break registry integrity if unchecked |
| Springdoc/OpenAPI → generated client | contract drift can cause the SPA to submit or parse the wrong shape |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-03 | T | `StudentService#createStudent` / `updateStudent` | mitigate | Validate `parentId` server-side for presence and real IDs before saving; never trust client-side form validation alone. |
| T-02-04 | E | `ParentService#archiveParent` / `deleteParent` | mitigate | Use repository-backed active-student checks so an authenticated user cannot archive or delete a responsável that still anchors active students. |
| T-02-05 | I | generated Kubb contracts | mitigate | Regenerate OpenAPI + Kubb in the same plan and verify the generated DTOs expose the exact relationship fields the SPA will read and write. |
</threat_model>
